Multi-Digest Hashing: Empirical Cost Analysis

Daniel Berry
April 2026
Companion to: Compliance-First Data
Processing Architecture

A reasonable objection to the five-algorithm hash set proposed in Compliance-First Data Processing Architecture (Berry, 2026, Appendix C) is that computing five cryptographic digests per item imposes substantial cost at scale. This note provides empirical data to quantify that cost across different workload profiles, so the trade-off can be evaluated on evidence rather than intuition.

Test Environment

CPU: AMD Ryzen Threadripper 7960X (24-core, 4.2 GHz)
RAM: 128 GB DDR5  |  Storage: NVMe SSD
Software: Python 3.13 / OpenSSL (SHA-NI hardware acceleration)
Method: In-memory data, 64 KB chunks, 5 rounds, median reported. Single-pass architecture: one read, five hash instances updated per chunk.

The raw overhead is approximately 9x. The single-pass architecture eliminates redundant I/O, but the CPU cost of five hash functions is real. On NVMe storage (3-7 GB/s sequential read), all-five throughput of 0.25 GB/s on a single thread is CPU-bound, not I/O-bound.

What the 9x Means in Practice: It Depends on the Workload

The practical impact depends entirely on what else the pipeline does:

Pure hashing
Forensic imaging, triage, hash verification
Hashing
~100% / 9x overhead
Extract + hash
Container extraction, metadata, no OCR
Hash
~35% / ~3x overhead
Full eDiscovery
Extraction, OCR, indexing, classification
~8% / ~7% overhead

For pure hashing workloads, multi-digest cost is the entire cost and the 9x multiplier applies directly. SHA-256-only is the defensible default unless specific regulatory requirements justify additional algorithms.

These are hash-only, single-thread projections. Multi-worker engines multiply throughput: 8 workers processing different items achieve ~2 GB/s aggregate for all-five, comparable to single-thread SHA-256-only. Pipeline concurrency further masks the cost when hashing overlaps with extraction of other items.

Conclusion

The CPU cost of five-digest hashing is real and measurable, not negligible. For hashing-dominant workloads, the overhead is substantial; for full processing pipelines, it represents roughly 7% of total throughput. The correct engineering response is configurable, auditable mode selection: forensic mode (all five) when regulatory defensibility warrants it; non-forensic mode (SHA-256 only) when it does not. The processing manifest records the decision, making it reviewable and defensible in either direction.
