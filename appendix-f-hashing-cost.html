Appendix F: Empirical Multi-Digest Hashing Cost

Appendix C presents the architectural case for multi-algorithm hashing and argues that a single-pass implementation amortizes I/O cost across all digest computations. A reasonable objection, raised during peer review, is that computing five cryptographic digests per item imposes substantial processing overhead at scale. This appendix provides empirical data to quantify that cost.

F.1 Methodology

All measurements were taken on a single workstation: AMD Ryzen Threadripper 7960X (24 cores, 4.2 GHz boost), 128 GB DDR5 RAM, NVMe storage. Python 3.13 with hashlib (delegating to OpenSSL's C implementations, with SHA-NI hardware acceleration for SHA-1 and SHA-256). Data was generated in memory (os.urandom) to isolate CPU hash throughput from storage variability. Each measurement: 5 timed rounds after 2 warmup rounds, median reported. The single-pass architecture reads data in 64 KB chunks, updating all five hash instances per chunk, exactly as described in Appendix C.1.

The benchmark script and raw JSON results are published alongside this paper for reproducibility.

Measured values align closely with published benchmarks for OpenSSL on Zen 4. The hardware-accelerated algorithms (SHA-1, SHA-256) outperform the software-only algorithms (MD5, SHA-512, SHA3-256) by 2-4x, confirming that the legacy practice of computing "only MD5 for performance" is both insecure and slower than SHA-256 on modern hardware.

F.3 Single-Pass All-Five Cost

The following table reports the median time to hash data of each size using SHA-256 alone versus all five algorithms in a single pass.

Data Size Representative Item SHA-256 Only All 5 (Single Pass) Added Time Overhead
1 KB Short email, metadata 1 µs 8 µs +7 µs negligible in absolute terms
64 KB Typical email 24 µs 239 µs +215 µs +0.2 ms per item
1 MB Office document, PDF 0.4 ms 4.0 ms +3.6 ms ~9x SHA-256 alone
16 MB Large spreadsheet 7 ms 64 ms +57 ms ~9x SHA-256 alone
64 MB Small PST segment 27 ms 265 ms +238 ms ~9x SHA-256 alone
256 MB Large container chunk 113 ms 1,015 ms +902 ms ~9x SHA-256 alone

The raw hash overhead is real. Computing all five digests takes approximately 9x longer than SHA-256 alone. The single-pass architecture eliminates redundant I/O reads, but it does not eliminate the CPU cost of running five hash functions. On NVMe storage (3-7 GB/s sequential read), the all-five throughput of 0.25 GB/s means hash computation, not I/O, is the bottleneck. This corrects the theoretical analysis in Appendix C.1, which assumed storage throughput would mask the CPU cost; that assumption held for spinning disk (150-200 MB/s) and SATA SSD (500 MB/s) but does not hold for NVMe.

F.4 Pipeline Context: Why the 9x Multiplier Misleads

The 9x overhead applies to hash computation in isolation. In a real processing pipeline, hashing is one step among many. A processing engine extracting, indexing, and analyzing documents spends its time roughly as follows:

Extraction
Text & metadata
Indexing
Hash (5)
Other
Container extraction
Text & metadata
Indexing
Hashing (all 5): ~8%
Classification, dedup, other

For a representative 1 MB document processed at ~20 items/second (a typical rate for mixed-format eDiscovery workloads including extraction, text analysis, and metadata normalization), total per-item processing time is approximately 50 ms. The all-five hash adds 4 ms. Hashing accounts for roughly 8% of total per-item time. SHA-256 alone would account for less than 1%. The marginal cost of four additional digests is therefore approximately 7 percentage points of total pipeline throughput.

For OCR-heavy workloads (0.5-5 items/sec), hashing drops below 2% of per-item time. For simple items (emails, plain text at 50-200 items/sec), per-item hash time is sub-millisecond regardless of algorithm count.

These are hash-only projections on a single thread. They represent the ceiling, not the expected wall-clock impact, for three reasons:

  1. Multi-worker processing. Production engines distribute work across multiple worker threads. With 8 workers, aggregate hash throughput rises to ~2 GB/s for all-five, comparable to single-thread SHA-256-only. The per-item latency is unchanged, but job completion time scales with worker count.
  2. Pipeline concurrency. In a pipelined architecture, hash computation overlaps with extraction and indexing of other items. The hash step rarely sits on the critical path unless the workload is dominated by very large, pre-extracted files.
  3. Hash time as fraction of total. A 1 TB processing job that takes 20 hours end-to-end (a typical mixed-format run) would see roughly 1.0 hour of additional hash time for all-five versus SHA-256-only: approximately 5% of total wall-clock time.

F.6 Findings

Summary

The CPU cost of computing five digests versus one is real and measurable: approximately 9x for the hash computation step in isolation. For NVMe-class storage, the theoretical I/O-masking argument in Appendix C overstates the amortization; hash computation is the bottleneck, not storage read throughput.

However, hashing represents roughly 8% of total per-item processing time in a full extraction-and-indexing pipeline. The end-to-end throughput impact of choosing five algorithms over one is approximately 7 percentage points, not the 9x multiplier that isolated benchmarks suggest.

For engagements where even that overhead is unjustified, the architecture provides non-forensic mode (SHA-256 only) as a first-class configuration, with the mode selection recorded in the processing manifest. The decision of when five-algorithm hashing is warranted, and when it is not, is addressed in Section 6.

Three design conclusions follow from these measurements:

  1. Multi-digest hashing is a bounded cost, not a prohibitive one. The per-item overhead is measured in single-digit milliseconds for typical documents. At pipeline level, it is a single-digit percentage of total processing time. This cost is justified when the regulatory or evidentiary context demands algorithmic diversity and future-proofing (see Appendix C.3); it is not justified as a blanket default for all processing contexts.
  2. Mode selection is the correct engineering response. Rather than debating whether five hashes are always necessary or never necessary, the architecture treats algorithm selection as a configurable, auditable decision. Forensic mode computes all five; non-forensic mode computes SHA-256 only; custom configurations are supported for organizational policy. The critical requirement is that the processing manifest records which algorithms were computed, so that the absence of a hash value is never ambiguous.
  3. The I/O-masking argument requires qualification. Appendix C.1's claim that multi-digest hashing is "I/O-bound regardless of how many digests are computed" was accurate for spinning disk and SATA SSD. On NVMe storage, the combined hash throughput (~0.25 GB/s, single-thread, all five) falls below typical sequential read speeds (3-7 GB/s), making the operation CPU-bound. The architectural recommendation remains sound (single-pass is strictly better than multi-pass), but the performance characterization should reflect current storage hardware: the marginal CPU cost of additional digests is real, bounded, and justified by its regulatory purpose.
Acknowledgment. This empirical study was prompted by feedback from Harlan Carvey, who observed that five-algorithm hashing imposes non-trivial computational cost and is not standard practice in most private-sector digital forensics engagements. That observation is correct, and the data above quantifies the cost he identified. The architecture's response is not to dismiss the cost but to make it a conscious, documented, and configurable decision.
